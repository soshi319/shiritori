# しりとりアプリ

React（Vite, TypeScript, Tailwind CSS）と Deno（WebSocket）で作った、しりとりアプリです。
ふつうのしりとり（練習モード）とオンライン対戦バトルモード、CPU対戦バトルモードの3つのモードがあります。

## デプロイ先URL

- フロントエンド（動作確認用）： https://shiritori-beta.vercel.app/
- バックエンド（Render）： https://shiritori-backend.onrender.com/

※ 以前はバックエンドをDeno Deployで運用しており、マルチリージョン仕様の影響でマッチング処理が不安定になることがありましたが、単一インスタンスで常時稼働するRenderに移行したことで解消しました（無料プランのため、15分アクセスが無いとスリープし、次のアクセス時に起動まで30〜60秒かかることがあります）。

## モード構成

### 1. ふつうのしりとり（練習モード）

課題で要求されている基本仕様を満たす、1人用の練習モードです。バトルモードの複雑なルール（拗音の扱いなど）を、この画面で確認・練習できます。

### 2. バトルモード（オンライン対戦）

WebSocketを使った、リアルタイムのオンライン対人対戦モードです。練習モードのしりとりルールをベースに、4人のキャラクター、ダメージ計算、カウンター、状態異常などのRPG要素を追加しています。

### 3. バトルモード（CPU対戦）

対人戦の相手が見つからない時でも1人で遊べる、CPU対戦モードです。バトルモードのルール・ダメージ計算・必殺技・毒・コンボなどはすべてオンライン対戦と共通のロジック（`shared`ディレクトリ）をそのまま利用しているため、数値や判定に差はありません。

CPUの受け答えは、あえて辞書全体からのランダム抽選にせず、**音（ひらがな1文字）ごとに3〜4語を登録した固定辞書**（`shared/data/cpuDictionary.ts`）から順番に答えを返す方式にしています。

- 同じ音で連続して攻め続けると、CPUは持っている単語を使い切り、最終的に「ん」で終わる単語を口走って自滅する
- ただしCPU自身のHPが低い状態で、ちょうど4文字の「ん」終わり単語が残っていれば、あえてそれを選んで必殺技（一閃）を狙いにいく

サーバー側（オンライン対戦時にCPUと戦う場合）・フロントエンド側（オフラインでCPUと戦う場合）のどちらの実装も、この辞書を共通で参照しています。

## 実装した機能

### 必須仕様（練習モードで実装）

- 直前の単語を画面に表示
- 任意の単語を入力できる入力欄
- 直前の単語の末尾と、入力した単語の先頭を比較し、一致する場合のみ単語を更新／不一致の場合はエラー表示
- 末尾が「ん」で終わる単語が入力されたらゲーム終了
- 過去に使用した単語が入力されたらゲーム終了
- ゲーム中・終了後いつでも押せる、最初からやり直せるリセット機能

### 任意機能

1. **最初の単語をランダムに決定**：候補となる単語リストからランダムに選び、毎回違う単語からスタートする
2. **ひらがな以外**：カタカナ・絵文字・記号・空文字などを正規表現で弾き、しりとりとして不適切な入力を防ぐ。
3. **実在しない単語の判定**：JMdictをベースに独自生成した軽量辞書を使い、辞書に存在しない単語の入力を防ぐ。ただしゲームを強制終了させず、警告表示のみでその場での再入力を促す。
4. **オンライン対戦バトルモード**：練習モードのしりとりルールをベースに、WebSocketによるリアルタイム対人対戦モードを実装。詳細は次項「バトルモードの実装」を参照。
5. **CPU対戦バトルモード**：対人戦の相手がいなくても遊べるよう、固定辞書ベースのCPUと対戦できるモードを実装。詳細は「モード構成」を参照。
6. **今つながる過去に打った単語の表示**：過去に打った単語のなかで、現在つながっているので打てるが打つとゲームオーバーになる単語を表示する。

### バトルモード独自の実装

- **キャラクター選択**：4人のキャラクター（攻撃力・減衰率・固有スキルがそれぞれ異なる）から1人を選んで対戦
- **ダメージ計算**：`ダメージ = 攻撃力 × 減衰率^(文字数-1)` という指数関数的な計算式により、短い単語ほど大ダメージになる仕様
- **反射（カウンター）**：攻撃側の単語が、防御側が事前に予測していた単語と一致した場合、ダメージが攻撃側自身に跳ね返る
- **状態異常（毒）**：濁点・半濁点を含む単語を使うと、相手に毒を付与。重ね掛けが可能で、スタック数に応じて継続ダメージが増加する
- **コンボシステム**：同じ文字数の単語を連続して打つと、ダメージが段階的に上昇する
- **必殺技（ばくだん）**：HPが一定以下の状態で、4文字かつ「ん」で終わる単語を打つと、大ダメージの必殺技が発動する
- **先攻・後攻とハンデ**：先攻・後攻をランダムに決定し、後攻側には一時的なHPボーナス（オーバーヒール）を付与
- **拗音・長音・カタカナ・濁点の正規化**：公式ルール（第4条）に基づき、拗音（例：「しゃ」）で終わった場合は次の単語もその2文字から始める必要がある、といった細かいルールに対応
- **CPU対戦**：対人戦と同じルール・ロジックを使いつつ、CPUの受け答えのみ固定辞書ベースの独自ロジックに差し替え（詳細は「モード構成」を参照）

## 実装したデザイン

- Tailwind CSSによる、ダークテーマを基調としたUI
- HPバーの被弾演出（ダメージを受けた際、緑のバーとオレンジの残像バーが時間差で減っていく二重バー表現）
- オーバーヒール（後攻ボーナス）時、通常のHP分は緑、超過分は水色で継ぎ足して表示するHPバー
- ヒット・反射・コンボ・毒付与など、状況に応じた専用のポップアップ演出（SVG/CSSアニメーション）
- 先攻・後攻が決まった瞬間の発表演出画面
- スマートフォンでのプレイを想定し、`100dvh`やスワイプ操作に対応したレイアウト

## 技術スタック

- **フロントエンド**：React（Vite）、TypeScript、Tailwind CSS
- **バックエンド**：Deno、WebSocket（Deno標準API）
- **通信**：WebSocket（対戦・マッチング）、HTTP API（辞書チェック）
- **辞書データ**：大元のJMdictから古語やマニアックな希少語を除外した「しりとり用軽量辞書（JSON）」を事前ビルドし、メモリ上に展開して利用
- **型・ロジックの共有**：`shared`ディレクトリを設け、フロントエンド・バックエンドの両方から同じ型定義（メッセージ型、キャラクターデータ）やロジック（ダメージ計算、しりとり判定、CPU用辞書）を参照する構成にし、二重管理によるズレを防止

## AIをどのように活用したか

**AIの活用に関するスタンス**
今回は、自身で考案した「オンライン対戦バトルモード」のアイデアを、期間内に実際に動く形として実装しきることを最大の目標としました。しかし、Reactに触れるのは今回が全くの初めてであったため、WebSocket通信や複雑な状態管理といった難易度の高い要件を、ゼロから完全に理解して実装するには時間が足りませんでした。そこで、コードは基本的にAIに書いてもらう方法を取りました。これにより、実装スピードとアプリケーションの完成度を優先させました。そのため現時点では、AIが生成したコードの裏側にある深い仕組みを、自分自身で完璧に理解できているわけではありません。しかし、まずは自分の思い描いたゲームを一通り作り上げるという最大の目標は達成できました。今後は、今回完成させたこのコードを参考に、別のコードを書きながら、ReactやWebフロントエンド技術への理解を深めていきたいと考えています。

### Google Gemini 

- 開発の初期段階で、ゲームのアイデア（しりとり×対戦システムのコンセプト、キャラクター設定の方向性など）を整理するブレインストーミングの相手として使用
- 実装中にエラーが発生した際、その都度エラー内容を伝えて原因を確認する用途で使用

### Claude (Sonnet)

設計から実装、デバッグまで、対話を通じて次のように活用しました。

- **設計の壁打ち** ディレクトリ構成、Reactフックの使い分け、`shared` ディレクトリを用いた型・ロジックの一元管理など、実装前の方針決定。
- **独自ルールの具体化** 「反射」「毒」などの曖昧だったアイデアを、対話を通じて発動条件やダメージ計算式に落とし込み。
- **バグの早期特定・修正** 初学者には気づきにくいReact特有のバグ（stale closure等）やロジックの誤りを、症状から逆算して特定。
- **CPU対戦モードの実装** オンライン対戦と同じ`shared`ロジックを再利用する形でのCPU対戦モードの設計・実装、およびCPU用固定辞書の作成・不具合修正。
- **技術概念の個別学習** 自分の理解度（チュートリアル修了直後）に合わせて、TypeScriptの型やReactの概念を都度解説してもらいながら実装。
- **インフラ課題の調査・移行作業** Deno Deployのマルチリージョン仕様によるマッチング問題の原因切り分けと、代替ホスティング（Render/Railway/Fly.ioの比較）の検討、Renderへの移行作業、およびメモリ不足エラーを解決するための軽量辞書生成スクリプトの実装。
- **ドキュメントやアプリ内文章の作成** このREADMEを作成する際、自分の開発への意図や現状の課題を、正確に伝えるための壁打ち相手・文章の推敲として活用。また、アプリ内での説明文の作成に使用。
